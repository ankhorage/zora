# AGENTS.md

## Scope

This file applies to the whole `ankhorage/zora` repository.

ZORA is the app-facing React Native / React Native Web UI kit for Ankhorage and other consumers. It provides reusable components, layout primitives, patterns, theme helpers, and static component metadata. Keep ZORA useful as a standalone UI/design-system package; do not couple it to a generated app, Studio, scanner product flows, nutrition/product lookup, or any specific runtime host.

## Repository facts

- Package name: `@ankhorage/zora`.
- Runtime: Bun.
- Language: TypeScript, ESM, strict mode.
- Renderer target: React Native and React Native Web.
- Main source root: `src/`.
- Build output: `dist/`.
- Public root export: `src/index.ts`.
- Public metadata export: `src/metadata/index.ts`.
- README is generated by Paradox. Do not edit `README.md` manually; update source/docs and regenerate where applicable.

## Core architectural rules

### ZORA owns app-facing UI

ZORA components are the app-facing design-system surface. They should be reusable outside Ankhorage internals and should not depend on generated-app assumptions.

Good:

- `Text`, `Heading`, `Button`, `Card`, `ProductCard`, `BarcodeScannerView`
- generic props
- generic event names
- static metadata describing props, slots, events, and requirements

Avoid:

- app-specific business logic
- template-specific copy baked into components
- generated-app routing assumptions
- direct Studio-only concerns
- product/nutrition lookup logic
- scanner result lookup or persistence logic

### Surface stays lower-level

`@ankhorage/surface` is a lower-level render foundation. ZORA may build on it internally, but consumers should generally import app-facing primitives from `@ankhorage/zora`, not directly from Surface.

When adding or changing public app UI, prefer a ZORA component/API rather than asking downstream apps to import from Surface.

### No runtime or platform coupling

ZORA components must stay runtime-adapter-neutral.

Do not import:

- Expo native APIs such as `expo-camera`, `expo-location`, `expo-media-library`, etc.
- `@ankhorage/runtime`
- generated app code
- Studio internals
- app-specific stores or routers

Instead, expose adapter-neutral props and static metadata. Platform packages/runtime adapters can inject native elements or handlers.

Example: `BarcodeScannerView` owns the visible scanner shell and accepts a `camera` slot/prop. The actual native scanner belongs to an app/runtime adapter such as an Expo adapter.

### Static requirements live in metadata

If a component needs platform permissions or capabilities, express them as static metadata through `requirements`.

Example concepts:

- camera permission
- barcode scanner capability

Keep requirements serializable and declarative. Do not implement native permission prompts inside ZORA components.

### Metadata is part of the public contract

Component metadata drives authoring/generation in downstream packages.

When changing component props, events, slots, or requirements:

- update metadata in `src/metadata`
- keep prop names aligned with the actual component API
- keep event payload metadata aligned with emitted payloads
- add/update tests where metadata is covered
- add a changeset for public API or metadata changes

## Source organization

Common locations:

```txt
src/components/        reusable atomic and composed UI components
src/foundation/        layout/foundation primitives
src/layout/            app-shell and screen layout primitives
src/patterns/          higher-level reusable UI patterns
src/metadata/          static component metadata for generators/Studio/runtime
src/theme/             theme creation, provider, hooks, tokens
examples/              examples used by docs and smoke tests
paradox/               generated docs output
dist/                  build output
```

Keep new components close to the relevant category:

- generic low-level UI: `src/components/<component-name>/`
- layout structures: `src/layout/<layout-name>/`
- reusable product-agnostic patterns: `src/patterns/<pattern-name>/`
- theme-only concerns: `src/theme/`

## Public API rules

Public exports are explicit.

When adding a new public component:

1. create the component module
2. export component and public types from its local `index.ts`
3. export component and public types from `src/index.ts`
4. add static metadata if the component can appear in manifests/Studio
5. add/update tests
6. add a changeset if public API changes

When adding a metadata-only export:

1. update `src/metadata/*`
2. export through `src/metadata/index.ts`
3. verify `./metadata` package export still builds

Do not rely on deep imports from consumers. If consumers need it, export it intentionally.

## Component design rules

### Props

- Prefer clear, serializable, platform-neutral props.
- Use TypeScript exported prop types for public components.
- Keep defaults inside the component where they are generic and safe.
- Avoid booleans that encode business decisions. Prefer small enums for meaningful variants.
- Avoid passing raw native event objects as public API unless absolutely necessary.
- Event payloads should be normalized to plain serializable objects.

### Events and actions

Components may expose callback props such as `onPress`, `onManualEntry`, `onBarcodeScanned`, etc.

Callbacks should receive generic payloads:

- scanner events should emit normalized scan payloads
- form events should emit values/errors where applicable
- no app-specific side effects

Do not invoke generated-app actions directly from ZORA. The runtime/action layer handles callback-to-action mapping.

### Children and slots

- Use `children` for generic content composition.
- Use named props/slots only when the component has a stable pattern need.
- Reflect slot behavior in metadata when the component is manifest-authorable.

### Styling

- Use ZORA theme/tokens and existing foundation primitives.
- Avoid hardcoded brand-specific styling.
- Avoid web-only CSS assumptions.
- Keep React Native and React Native Web compatibility.
- Avoid platform-specific forks unless they are strictly visual and safe.

### Accessibility

- Preserve labels, roles, and touch targets where existing primitives support them.
- Do not remove accessible text just to make UI shorter.
- Make interactive controls keyboard/screen-reader reasonable for web where possible.

## Metadata rules

When defining metadata:

- `name` must match the manifest/component type.
- `category` should be one of the existing metadata categories.
- `directManifestNode` should be true only for components intended as manifest nodes.
- `props` must match actual public props.
- `events` must match actual callback props and payloads.
- `requirements` must be declarative and platform-neutral.
- `allowedChildren` and `slots` should describe real composition constraints.
- `blueprint.defaultProps` should be safe, generic, and useful in Studio.

Do not put runtime implementation details into metadata.

Good:

```ts
requirements: {
  permissions: [{ permission: 'camera' }],
  capabilities: [{ capability: 'barcodeScanner' }],
}
```

Bad:

```ts
requirements: {
  packages: ['expo-camera'],
  adapters: ['ExpoBarcodeScannerAdapter'],
}
```

The latter belongs to a platform runtime package, not ZORA.

## Scanner-specific rules

`BarcodeScannerView`, `CameraPermissionView`, and `ScanOverlay` are generic scanner UI patterns.

They must remain:

- camera-adapter-neutral
- permission-client-neutral
- product-lookup-neutral
- template-neutral

Do not add:

- product database lookup
- nutrition API logic
- barcode vendor assumptions beyond generic labels/copy
- Expo camera imports
- app navigation side effects

Allowed:

- visible scanner layout
- permission-state UI
- manual-entry UI
- generic scan overlay
- generic callback props
- static permission/capability metadata

## Theme rules

- Use `ZoraProvider` at app root in examples.
- Use `useZoraTheme` for theme access.
- Prefer semantic theme tokens over raw colors.
- Do not hardcode app-specific colors.
- Keep light/dark compatibility in mind.
- New theme APIs require tests and a changeset.

- `README.md` is generated by Paradox; do not edit it manually.

## Changesets

Add a changeset for any public-facing change:

- new component export
- prop/type change
- metadata change used by Studio/generators
- theme API change
- behavior change in an exported component
- dependency/public peer requirement change

Use patch unless the change is clearly minor/major.

No changeset is usually needed for:

- internal-only refactor
- test-only changes
- docs-only changes that do not affect published package content
- non-public implementation cleanup

When unsure, add a patch changeset.

## Dependency rules

Current direct dependencies include:

- `@ankhorage/color-theory`
- `@ankhorage/contracts`
- `@ankhorage/surface`

Peer dependencies include React Native/Expo-related packages. Keep optional platform peers optional when they are only needed for specific components.

Do not add large runtime dependencies casually. For new dependencies:

- justify why ZORA needs it directly
- prefer peer dependencies for host/platform concerns
- avoid bringing generated-app/runtime concerns into this package

## Documentation rules

- Do not manually edit generated `README.md`.
- Update examples/source docs used by Paradox when changing docs.
- Run `bun run docs` if docs output should be refreshed.
- Keep examples generic and useful outside Ankhorage-specific templates.
- Avoid copy that makes ZORA sound usable only inside Ankhorage generated apps.

## Working style for agents

Before coding:

1. inspect the existing component/category pattern
2. inspect current exports
3. inspect related metadata
4. identify whether a changeset is needed

While coding:

1. keep changes narrow
2. preserve public API compatibility where possible
3. do not introduce app/runtime/platform coupling
4. update tests with behavior
5. update metadata together with component changes

Before final handoff:

1. run validation or clearly report what was not run
2. include `git diff --stat`
3. mention public API/metadata changes
4. mention changeset path
5. mention any follow-up issue needed

## Common pitfalls

- Editing generated `README.md` manually.
- Adding Expo/native imports into ZORA components.
- Adding app-specific lookup/business logic into patterns.
- Changing component props without metadata updates.
- Changing metadata without tests.
- Forgetting root `src/index.ts` exports.
- Forgetting `src/metadata/index.ts` exports.
- Hiding UI in Studio/runtime because action props are not normalized downstream.
- Treating ZORA as generated-app-only instead of standalone UI kit.
- Importing Surface directly from consumers when a ZORA-facing component should exist.

## Current architectural direction

ZORA should own first-class app-facing UI primitives and structured components.

Surface remains the lower-level foundation. Runtime/platform packages adapt ZORA components to concrete hosts. Studio consumes ZORA metadata and components for authoring. Generated apps render ZORA components through the runtime registry.

Keep this separation:

```txt
ZORA
  app-facing UI + static metadata

Surface
  low-level render foundation

Runtime
  manifest rendering, registry, action/data binding

Platform runtime packages
  Expo/native adapters, permission/capability planning, config plugins

Studio
  authoring product, preview decoration, panels, selection

Generated apps
  output consumers of manifests, runtime, ZORA, and platform packages
```
